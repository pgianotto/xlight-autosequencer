# Phase 1 Data Model: Tauri Desktop Packaging

**Feature**: 052-tauri-desktop-packaging
**Date**: 2026-04-22

This feature is primarily a packaging concern — the existing data model (analysis JSON, library, themes, sections, etc.) is preserved unchanged. Only the new entities introduced by packaging are listed here.

---

## Runtime entities (live only for a session)

### SidecarProcess

Represents the PyInstaller-bundled Flask backend spawned by the Tauri shell.

**Fields**:
- `pid` (integer) — OS process ID of the sidecar.
- `port` (integer) — 127.0.0.1 port Flask bound to. Discovered via stdout handshake; not known at spawn time.
- `state` (enum) — one of `spawning`, `waiting_for_port`, `ready`, `exited`, `crashed`.
- `stdout_buffer` (string, transient) — lines read from sidecar stdout before `ready` state, used to extract port and surface backend log output.
- `exit_code` (integer, optional) — set when state transitions to `exited` or `crashed`.

**State transitions**:
- `spawning → waiting_for_port` — immediately after `Command::spawn()` returns.
- `waiting_for_port → ready` — on reading a line matching `^XLIGHT_BACKEND_PORT=(\d+)$` from stdout.
- `waiting_for_port → crashed` — on sidecar exit before the port line is read.
- `ready → exited` — on clean sidecar exit (e.g., explicit shutdown request).
- `ready → crashed` — on unexpected sidecar exit while `ready`.

**Lifecycle**: created when the Tauri window is shown; destroyed when the window is closed or the app quits (SIGTERM with 5-second grace period, then SIGKILL).

---

### BackendEndpoint (frontend-side)

The frontend-side representation of the sidecar's HTTP API, set once and used for every request for the lifetime of the app.

**Fields**:
- `base_url` (string) — `http://127.0.0.1:<port>`. Set from the `backend-ready` Tauri event payload.
- `ready` (boolean) — false until the Tauri event fires; true thereafter.
- `lost_at` (ISO timestamp, optional) — set if the sidecar crashes mid-session; triggers a reconnect UI.

**Visibility**: lives in a small Zustand store slice; every `fetch` call reads from it.

## Persistent entities (live on disk across sessions)

### PackagingManifest (build-time only)

Generated by the build pipeline; shipped inside `Contents/Resources/` of the `.app`. Read once at startup to identify the build.

**Fields**:
- `app_version` (semver string) — e.g., `"1.0.0"`.
- `build_timestamp` (ISO string) — when the `.app` was built.
- `target_arch` (enum) — `"aarch64"` or `"x86_64"`.
- `frontend_commit` (git sha) — commit of `src/review/frontend/` when built.
- `backend_commit` (git sha) — commit of the Python code when built.
- `bundled_vamp_plugins` (array of strings) — names of Vamp plugin packs bundled (for About/Credits view).
- `download_model_manifest_url` (string) — URL of a JSON listing known demucs model SHA256s, so that hash verification in weights download can be updated without an installer rebuild.

**Storage**: `Contents/Resources/packaging-manifest.json`, read-only, bundled at build time.

---

### ModelCache (new persistent directory)

On-disk location for large downloaded model weights.

**Path**: `~/Library/Application Support/XLight/models/`

**Structure**:
```
models/
├── torch-hub/
│   └── checkpoints/
│       ├── 955717e8-8726e21a.th   # htdemucs_6s shard 1
│       ├── <...>.th                # additional shards
│       └── <...>.th.sha256         # hash verification files
└── .download-state.json            # resume state for interrupted downloads
```

**Fields of `.download-state.json`**:
- `model_name` (string) — e.g., `"htdemucs_6s"`.
- `in_progress_shards` (array of objects) — each `{url, bytes_downloaded, expected_size, sha256}`.
- `completed_shards` (array of strings) — URLs of shards already verified.
- `last_updated` (ISO string).

**Reuse of torch hub path**: the torch hub cache structure is preserved so that demucs itself finds the weights without any config override — it is pointed at this directory via the `TORCH_HOME` env var set by the sidecar launcher.

---

### SandboxFallbackStems (conditional)

Stem cache path used when the source-audio-adjacent `.stems/<hash>/` location is not writable.

**Path**: `~/Library/Application Support/XLight/stems/<md5_hash>/`

**Structure**: identical to the existing `.stems/<hash>/` layout — `drums.wav`, `bass.wav`, `vocals.wav`, etc.

**Created lazily**: directory is only created on first fallback write; source-adjacent writes still happen when the source directory is writable (preserves current CLI behavior).

---

## Preserved entities (unchanged — listed for clarity)

These entities already exist in the codebase and are **not modified** by this feature. They remain in their current paths and formats:

- **Library index**: `~/.xlight/library.json` (existing)
- **Settings**: `~/.xlight/settings.json` (existing)
- **Custom themes**: `~/.xlight/custom_themes/*.json` (existing)
- **Custom variants**: `~/.xlight/custom_variants/*.json` (existing)
- **Logs**: `~/.xlight/logs/` (existing)
- **Analysis cache**: `<audio>_analysis.json` next to source audio (existing)
- **Stems (primary location)**: `.stems/<md5>/` next to source audio (existing, with new fallback above)

The shared-with-CLI decision (R7 in research) means the packaged app and the CLI read/write the same files. Any future migration of these paths is a separate concern and not in scope for this feature.

## Relationships

```
PackagingManifest (read-only, at build time)
    │
    └─→ read once at app startup by Rust shell
            │
            ▼
        SidecarProcess (spawned per app session)
            │
            ├─→ exposes BackendEndpoint to frontend (via Tauri event)
            │
            └─→ on stem-separation request:
                    │
                    ├─→ reads/writes ModelCache (torch hub format)
                    │
                    └─→ reads/writes SandboxFallbackStems when source-adjacent write fails
```

No new relational schema, no new database. All persistence is either JSON-on-disk (manifests, state files) or opaque binary files (model weights, stems) in well-defined directory layouts.
