# Data Model: Audio Analysis and Timing Track Generation

**Branch**: `001-audio-timing-tracks` | **Date**: 2026-03-22

---

## Entities

### AudioFile

Represents a loaded MP3 file and its metadata.

| Field | Type | Description |
|-------|------|-------------|
| `path` | string | Absolute path to the source MP3 file |
| `filename` | string | Base filename (no path) |
| `duration_ms` | integer | Total duration in milliseconds |
| `sample_rate` | integer | Sample rate in Hz (e.g., 44100) |
| `channels` | integer | 1 (mono) or 2 (stereo) |

**Notes**:
- Audio is loaded as mono float32 numpy array internally (stereo is mixed down).
- Loaded once and shared across all algorithm runs — no redundant file reads.

---

### AnalysisAlgorithm

Describes one algorithm run configuration. Stored in the output to enable
full reproducibility.

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Unique algorithm identifier (e.g., `"qm_beats"`, `"librosa_beats"`) |
| `element_type` | string | One of: `beat`, `bar`, `onset`, `frequency`, `percussion`, `harmonic`, `melody`, `structure`, `tempo` |
| `library` | string | Library name and version (e.g., `"librosa 0.10.1"`, `"vamp + qm-vamp-plugins 1.7.1"`) |
| `plugin_key` | string or null | Vamp plugin key used (e.g., `"qm-vamp-plugins:qm-barbeattracker:beats"`). null for non-Vamp algorithms. |
| `parameters` | object | Key-value pairs of algorithm-specific settings |

---

### TimingMark

A single timing event within a track.

| Field | Type | Description |
|-------|------|-------------|
| `time_ms` | integer | Timestamp in milliseconds from the start of the audio |
| `confidence` | float or null | Confidence score 0.0–1.0 if the algorithm provides it; null otherwise |

**Notes**:
- Timestamps are integers (milliseconds) to avoid floating-point serialization
  inconsistencies that would break determinism (see Decision 5 in research.md).
- Marks within a track are always sorted ascending by `time_ms`.

---

### TimingTrack

A named sequence of timing marks generated by one algorithm.

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Human-readable track name (e.g., `"qm_beats"`, `"drums"`) |
| `algorithm_name` | string | References `AnalysisAlgorithm.name` |
| `element_type` | string | Copied from the algorithm for convenience |
| `mark_count` | integer | Total number of timing marks |
| `avg_interval_ms` | integer | Average milliseconds between consecutive marks (rounded) |
| `quality_score` | float | Combined score 0.0–1.0 for lighting usefulness (density + regularity). See research.md Decision 8. |
| `marks` | array of TimingMark | The actual timing data |

**Derived fields** (`mark_count`, `avg_interval_ms`, `quality_score`) are computed at
serialization time from the `marks` array — they are never stored as source-of-truth values.

---

### AnalysisResult

The complete output of a single analysis run. This is what gets written to the
output JSON file.

| Field | Type | Description |
|-------|------|-------------|
| `schema_version` | string | JSON schema version (currently `"1.0"`) |
| `source_file` | string | Absolute path to the analyzed MP3 |
| `filename` | string | Base filename |
| `duration_ms` | integer | Duration of the analyzed audio in ms |
| `sample_rate` | integer | Sample rate used for analysis |
| `estimated_tempo_bpm` | float | BPM estimated during analysis |
| `run_timestamp` | string | ISO 8601 UTC timestamp of when the analysis ran |
| `algorithms` | array of AnalysisAlgorithm | All algorithms that were run |
| `timing_tracks` | array of TimingTrack | All generated timing tracks |

## State Transitions

The pipeline has a linear state flow — no branching or retries at the data level:

```
MP3 File
  → AudioFile (loaded, metadata extracted)
    → [per algorithm] TimingMark[] (raw float timestamps → integer ms, sorted)
      → TimingTrack (marks + derived stats)
        → AnalysisResult (assembled, timestamped)
          → JSON file (serialized)
```

If an algorithm fails, its corresponding `TimingTrack` is omitted from
`AnalysisResult.timing_tracks` and the failure is logged to stderr. The run
continues with remaining algorithms.

---

## Export Subset (track selection)

When the user runs `xlight-analyze export --select beats,drums`, the tool:

1. Reads the existing `AnalysisResult` JSON.
2. Filters `timing_tracks` to only those whose `name` matches the selection.
3. Filters `algorithms` to only those referenced by the selected tracks.
4. Writes a new JSON file with the subset. All other fields (`schema_version`,
   `source_file`, `duration_ms`, etc.) are preserved unchanged.

The export operation does NOT re-analyze — it only filters the existing result.
